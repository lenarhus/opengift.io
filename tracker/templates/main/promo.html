Процесс работы

Процесс поиска и подбора исполнителя занимает не больше минуты и состоит из пары простых шагов..

  • Шаг 1

    Создание задачи

    Удобный инструмент создания задач позволяет быстро описать в удобном формате список необходимых работ и следить за их исполнением в режиме реального времени.

  • Шаг 2

    Рассылка приглашений

    Heliard автоматически подберет исполнителей для ваших задач. Наш обучаемый алгоритм ищет среди исполнителей только тех, кто лучше всего справиться с вашими задачами и готов начать работать в ближайшее время.

  • Шаг 3

    Выбор исполнителя

    Вы можете выбрать из предложенных вам исполнителей одного или нескольких после обсуждения заданий. Выбранные исполнители будут добавлены в ваш проект и в дальнейшем могут работать с вами постоянно.

  • Шаг 4

    Закрытие задачи

    Вы сами решаете, когда задача выполнена в полном объеме. В процессе работы вы можете обсуждать с исполнителем все детали в отдельной ветке обсуждения, туда же вы можете скидывать необходимые файлы или скриншоты.

Общие постулаты


1. Разработчик должен своевременно и в полном объеме запросить данные, необходимые для выполнения задачи (доступы, файлы и т.д.) на этапе ее оценки

2. Разработчик должен самостоятельно проверять результат выполнения задачи перед установкой статуса «на проверку», а также работу функционала, прямо или косвенно связанного с тем, в которые он вносил изменения

3. Все сведения об изменениях кода должны вноситься в хранилище версий в рамках той задачи, которая в данный момент выполняется

4. По всем задачам, предполагающим серьезное изменение/дополнение функционала должна присутствовать документация в вики проекта

5. Разработчик должен всячески избегать нанесения ущерба работоспособности уже существующим системам проекта. Если в процессе работы был нанесен ущерб работоспособности существующему функционалу и потребовалось время на исправление, работа считается принятой лишь частично.

Общий регламент фронт-энд разработки


1. Не рекомендуется использование в CSS и JS селекторов помимо классов (исключение составляют семантически обязательные наследники: table.class1 td, ul.class2 li, select.class3 option)

2. Если необходимо в рамках существующей верстки добавить класс для взаимодействия с JavaScript, его наименование должно быть согласно шаблону js-действиеОбъект

3. Не рекомендуется использование !important

4. Отделение логических блоков производится по типу наследования в названии класса.

Пример:

div.userInfo
span.userInfo__name
div.userInfo__rating
a.userInfo__rating_vote

5. Наименования переменных в JS назначаются по шаблону типОбъект(Действие/Свойство)

Пример:

Boolean bLogoutFlag
Integer iStepCounter
String sUserName
Function buttonClick
jQueryObject $buttonLogout
Object oQueryData
Array(List) aUsers

6. Шаблоны HTML и селекторы должны присутствовать только в конфигурационных переменных

Примеры:

var sUserInfoTpl = ‘<div class=”userInfo”>#NAME# #LAST_NAME#</div>’, sUserPageSelector = ‘.userPage’;

sUserInfoTpl.replace(‘#NAME#’, oUser[‘NAME’]); sUserInfoTpl.replace(‘#LAST_NAME#’, oUser[‘LAST_NAME’]);
$( sUserPageSelector).html(sUserInfoTpl);

7. Все ключи конфигурационных объектов пишутся заглавными буквами:

Пример:

oUserQueryConfig = {
‘NAME_TPL’: ’<div class=”userInfo-name”></div>’,
‘SESSION_ID’: sessid,
‘QUERY_URL’: ’/get_user_info’
}

8. При верстке необходимо обязательное следование стандартам W3C

9. Все пути к изображениям и ссылки на текущий ресурс должны быть относительными за исключением использования CDN

10. Стили и скрипты обязательно должны быть вынесены в соответствующие CSS и JS файлы за исключением моментов, когда для них необходимы параметры из бэкенда. Параметры в этом случае выносятся в конфигурационные JS-объекты, а сама логика в любом случае выносится в JS-файл.

11. Блоки верстки должны предусматривать изменение размеров изображений и длины текстовых полей, на которые может влиять пользователь.

12. Все кнопки в верстке обязательно должны визуально реагировать на наведение курсора.

13. Для всех полей ввода обязательно должны быть предусмотрены классы, которые появляются при ошибочном вводе данных. Все формы должны предусматривать JS-валидацию.

14. Все ajax-запросы, требующие ожидания пользователя должны сопровождаться визуальным эффектом загрузки (gif, css или apng лоадер)

15. Поля ввода, где заранее известен формат ввода должны обязательно сопровождаться маской ввода (телефон, СНИЛС и так далее)

16. Поля ввода должны иметь корректный tabIndex

17. Поля ввода, в которых предусмотрено удаление введенных данных (например, динамический поиск), должны сопровождаться крестиком очистки введенной информации.

Общий регламент бэкенд-разработки


1. Имена классов, функций и значимых переменных всегда должны быть "говорящими" для того, чтобы читатель мог сразу понять для чего они используются.

2. Имя метода или функции должно явно говорить обо всем, что она делает, при этом имя должно обобщать все небольшие действия для сохранения лаконичности. Если функция производит два действия, которые нельзя обобщить, они оба должны присутствовать в названии.

Верно:

getName()
fillAndPrintOrder()

Не верно:

process()
strCmp()

3. Необходимо неуклонно следовать принципу DRY

4. Имена переменных, классов и методов должны быть на английском языке без использования транслита русских слов.

5. В рамках одного проекта для одного языка допускается следование лишь одной нотации (camelCase, PascalCase и тд)

6. Константы, аббревиатуры и строковые ключи массивов должны всегда печататься заглавными буквами.

7. Не рекомендуется использовать в коде числа, кроме 1 и 0. В случае необходимости использования чисел все они должны быть определены в константах или конфигурационных файлах.

8. Не рекомендуется использовать в условиях и операторах сравнения кириллический текст.

Не верно:

if ($element[‘NAME’] == ‘Нечто’) {
$isRequired = $fieldText != ‘Обязательное поле’

Верно:

if ($element[‘CODE’] == ‘something’)
$isRequired = $field[‘IS_REQUIRED’] == ‘Y’;

9. Все переменные должны быть определены явно

10. К каждому методу должна присутствовать документация (PHPDoc, PyDoc)

11. Не рекомендуется использовать комментарии нигде кроме документации, вместо комментирования функционала предпочтительно создание метода с говорящим названием.

12. Не рекомендуются закомментированные участки кода в коммитах

13. Все параметры, получаемые от пользователя ($_POST, $_GET, и т.д.) должны быть принудительно приведены к нужному классу. Случайные символы и нулевой байт в строках должен экранироваться функцией trim()

Не верно:

$objectForSave->number = $_POST[‘number’]
$objectForSave->name = $_POST[‘name’]

Верно:

$objectForSave->number = intval($_POST[‘number’])
$objectForSave->name = trim($_POST[‘name’])

14. Все параметры, участвующие в создании SQL запросов должны быть экранированы специальными функциями (mysql_real_escape_string и тд)

15. Логика и представление всегда должны быть разделены и должны располагаться в разных файлах. Не допускается использование запросов к БД в файлах шаблонов.

16. Все операции, связанные с записью и удалением файлов, должны обязательно проверяться таким образом, чтобы исключить изменение файлов за пределами корневой директории проекта.

17. В пределах корневой директории проекта запись и удаление файлов должны быть разрешены только в те директории, которые используются для этих целей.

18. Типы записываемых или удаляемых файлов должны подвергаться жесткому контролю таким образом, чтобы избежать изменения файлов необходимых для правильной работы проекта.

19. Если назначение метода - поиск множества объектов и возврат их массива, но ни один объект не найден, то метод должен возвращать пустой массив и код, принимающий результат такого метода должен учитывать эту ситуацию.

20. При запросах к БД всегда должны обрабатываться ситуации получения результата, отличного от ожидаемого (например, пустого)

21. Максимальная длина одной строки должна составлять 120 символов

22. Сообщения об ошибках и об успехе действий пользователя должны присутствовать во всех ajax-запросах.

Что такое Heliard?


Heliard (Гелиард) — это платформа, посредством которой происходит подбор и контроль различного рода веб-специалистов (преимущественно веб-разработчиков, html-верстальщиков и веб-дизайнеров).

Кроме этого, Heliard является сообществом, ключевой задачей которого является обмен опытом, мнениями, ресурсами, а также помощь в обучении и повышении уровня знаний веб-специалистов.

С помощью нашей системы Вы можете не только найти исполнителей для своих проектов среди отобранного нами списка профессиональных веб-разработчиков, но и организовать работу в своей команде совершенно бесплатно!

Понятия и общие правила


1. Ключевые понятия


Заказчик — ключевая фигура; лицо, создавшее Проект.

По умолчанию Заказчик является одновременно Инвестором и Менеджером проекта. Однако, у него всегда есть возможность пригласить на эти роли любых других компетентных людей.

Менеджер проекта — представитель заказчика либо выбранный из числа работников Heliard специалист, имеющий право по созданию и контролю выполнения задач Исполнителями. Менеджерами проекта может быть неограниченное количество человек.

Участник — специалист, ответственный за выполнение и управление только своими задачами. Участником проекта может быть человек, назначенный как из своих, так и из числа специалистов Heliard. У проекта может быть неограниченное количество исполнителей.

Инвестор — лицо, осуществляющее оплату работ по Проекту. Все расходы по проекту списываются с его лицевого счета в Heliard. По умолчанию инвестором является Заказчик проекта. У проекта может быть только один Инвестор.

Гость — приглашенное в проект лицо. Количество гостей не ограничено.

Ответственный - лицо, которое может выполнять задачу и учитывать по ней время выполнения.

2. Проекты


Проект — это группа задач определенной тематики, в рамках которого будет происходить взаимодействие между Заказчиком, Менеджером проекта и Участниками.

Работа по проекту начинается с рассылки приглашений в проект потенциальным Участникам. С этого момента приглашенные в проект люди становятся Гостями проекта.

Видеть конкретные проекты могут только их авторы, менеджеры и приглашенные в проект Гости.

3. Задачи


Задача представляет собой ветку обсуждения какого-либо задания.

Работа по задаче начинается с оценки и обсуждения её выполнения с потенциальными исполнителями.

По результатам оценки и обсуждения задач Менеджер проекта выбирает Ответственного для задачи. Если ответственным выбирается гость проекта, с этого момента он становится Участником.

У задачи может быть несколько Ответственных, однако, единовременно вести работу по задаче может только один человек.

Возможность просматривать задачи имеют только автор задачи, её Ответственный, Менеджер проекта, а также приглашенные в задачу наблюдатели из лица Гостей и Участников.

3.1 Сообщения в задачах

У всех, приглашенных в проект людей, есть возможность оставлять Сообщения внутри доступных им задач.

Все сообщения в рамках тех или иных задач видны только Участникам и Менеджерам проекта. Гости видят только свои и адресованные им сообщения.

Личные сообщения видит только автор и адресат.

3.2 Статусы задач

Каждая задача имеет 4 статуса:

Не подтверждена (статус устанавливается только если Ответственный — платный специалист Heliard)
Не подтвержденную задачу нельзя начать делать. Для подтверждения задачи она должна быть оценена.

В работе
Этот статус означает, что исполнитель может приступать к задаче.

На проверке
Означает готовность задачи к проверке Менеджерами проекта.

Закрыта
При закрытии задачи производится списание денежных средств со счета Инвестора проекта и зачисление их на лицевой счет Ответственных, которые участвовали в задаче.

4. Работа со специалистами Heliard


Ответственные Heliard — это профессиональные работники, которые имеют цену часа и работают над задачами по Таймеру — счетчику количества потраченного на выполнение задачи времени.

Менеджеры Heliard — специалисты, которые помогают Заказчику ставить задачи исполнителям, руководят процессом разработки и получают деньги за закрытие задач, в которых они присутствуют как наблюдатели.

Платных специалистов в гелиарде можно выбирать по рейтингу и специальностям.
Рейтинг является качественным показателем опыта работника и высчитывается на основании закрытых им задач и допущенных ошибок.
Рейтинг также добавляется к часовой ставке специалиста. Специалисты с более высоким рейтингом являются более дорогими.

Порядок работы


1Заказчиком или Менеджером проекта создаются задачи.

Для того, чтобы время выполнения задачи было небольшим, нужно как можно более детально описать суть вопроса и требования к её выполнению.

2 Для приглашения специалистов Heliard Заказчик или Менеджер проекта должен выбрать из списка те задачи, которые необходимо сделать.

3После выбора задач, необходимо нажать кнопку «Пригласить специалистов» и в открывшемся окне вписать специальности, которые интересуют.

Это необходимо для того, чтобы из огромного количества специалистов система выбрала только тех, кто уже делал похожую работу.

4Приглашенные вами специалисты будут добавлены в проект в качестве Гостей. Они смогут видеть только задачи, в которые вы их пригласили.

Перед выбором также обратите внимание на график работы специалиста, он может быть на связи не каждый день. Кроме того посмотрите количество назначенных на него задач, его рейтинг, специализации и портфолио.

5После оценки времени выполнения задачи Ответственным задачи, Заказчик или Менеджер видят оценочную стоимость задачи в рублях.

Оценочная стоимость задачи — это оценка задачи в часах, умноженная на текущую ставку специалиста.

6После подтверждения Заказчиком или Менеджером проекта задачи, сумма замораживается на балансе Инвестора и не может использоваться при подтверждении других задач.

Для того, чтобы подтвердить выполнение задачи тем или иным Исполнителем, на счете Инвестора проекта должна быть достаточная для выполнения задачи сумма.

7Зарезервированная под выполнение задачи сумма будет будет списана с баланса Инвестора после закрытия задачи.

Стоимость задачи не может превысить выставленную специалистом заранее, но может быть меньше, если специалист справится быстрее.
Неизрасходованные Исполнителем средства возвращаются на баланс Инвестора.

5. Ответственность сторон


Заказчик должен своевременно и в полном объеме проверять задачи со статусом «На проверке».

В целях информирования, на почту, указанную Заказчиком при регистрации, будут приходить письма с напоминанием, а также соответствующие задачи будут выделены цветом.

Исполнитель оставляет за собой право приступить к выполнению задач по другим проектам, если Заказчик не выходит на связь более 2 двух дней с момента поступления вопросов от разработчика либо отправки задач на проверку.

Однако при этом Исполнитель не может оставлять проект и обязуется вернуться к нему не позднее двух дней после поступления информации от Заказчика

Задачи, которые находятся на проверке более недели, будут закрыты автоматически.

Исполнитель обязан качественно, без ошибок и в срок выполнять поставленные задачи, а также быть на связи в оговоренное с Заказчиком время.

При невыполнении требований, на Исполнителя будут накладываться санкции и штрафы.
